# AgentK v0.1 Target

AgentK v0.1 should make one claim concrete: a powerful AI agent can keep broad
tool access while high-risk boundary crossings still require policy,
provenance, and replayable evidence.

The release target is a credible reference MCP security shim, not a general
agent framework and not a production operating-system sandbox. A reviewer should
be able to put AgentK between an MCP client and server, run the same poisoned
scenario with and without the shim, inspect the trace, replay the evidence, and
understand why dangerous transitions were blocked.

## Release Shape

v0.1 is ready when the following are true:

- The one-command MCP killer demo shows poisoned MCP output attempting secret
  exfiltration and an unsafe file patch, with AgentK blocking both follow-up
  actions and writing a redacted trace.
- The MCP shim eval scorecard compares baseline passthrough with AgentK
  mediation and makes the improvement visible without reading the source.
- The subprocess stdio MCP proxy mediates tools, resources, and prompts through
  lifecycle checks, bounded IO, downstream shape validation, error redaction,
  explicit capability policy, trace output, and default-deny unsupported methods.
- `trace-inspect`, `replay`, `fork-replay`, and signature verification produce
  enough evidence for a reviewer to see policy decisions, missing capabilities,
  provenance, hash refs, and side-effect-free replay behavior.
- `cargo run --locked -- release-audit` passes locally and in CI, with only
  documented human-review warnings.
- Release docs clearly distinguish implemented shim behavior from deferred
  production custody, deployment, and sandboxing work.

## Better Than No AgentK

The core v0.1 proof is a before/after eval:

- Baseline passthrough lets poisoned MCP output request a risky downstream
  transition.
- AgentK mediation blocks the risky transition at the boundary.
- The denial names policy reasons and missing capabilities.
- The trace records hash-first evidence without dumping raw poisoned content,
  secrets, local paths, or private payloads.
- Replay confirms blocked side effects stay blocked.

If a skeptical reviewer cannot see this difference from a single local command
and the resulting trace, the release is not ready.

## Accepted v0.1 Limits

These are acceptable for v0.1 when clearly documented:

- The operator-ready path is the local subprocess stdio MCP proxy. Hosted,
  remote, or packaged production transport can remain future work.
- The static development signing key may remain for demos and CI, but strict
  release gates must support a configured signing key and must not claim
  production key custody.
- External secret references and metadata-only store checks may remain adapter
  boundaries. Production secret storage can remain future work.
- Real OS sandboxing, eBPF, cgroups, and filesystem isolation can remain future
  work.
- Resource subscription flows can remain unsupported unless they get an
  explicit policy contract; they must not become generic passthrough.

## Release Blockers

Do not tag v0.1 if any of these are true:

- CI or `cargo run --locked -- release-audit` fails.
- Unsupported MCP methods are forwarded as generic passthrough.
- Raw secrets, signing keys, `.agentk` logs, local paths, private traces, or
  credential-like values are tracked.
- Docs imply production key storage, production secret storage, complete hosted
  MCP transport, or real sandboxing before those exist.
- A blocked action cannot be explained from redacted evidence, policy reasons,
  and replay output.

## Autonomous Work Order

When master is clean and CI is green, prefer the next highest-impact item in
this order:

1. Close gaps that make the before/after MCP shim eval more convincing.
2. Harden the subprocess MCP proxy where it affects operator trust, redaction,
   lifecycle, bounded IO, or no-passthrough guarantees.
3. Add realistic MCP interop transcripts that cover tools, resources, prompts,
   notifications, and failure modes.
4. Improve evidence UX for `trace-inspect`, replay, fork replay, denial
   summaries, and signature checks.
5. Keep README, roadmap, threat model, MCP proxy docs, and release checklist
   aligned with implemented behavior and accepted v0.1 limits.

# v0.1 Accepted Limit Disposition

This document records which known limits are acceptable for AgentK v0.1 and
which ones would block a release. It is a release contract, not a promise that
these limits are good enough for production.

v0.1 is a credible local MCP security shim: a reviewer can run the shim between
an MCP client and a downstream stdio server, compare baseline passthrough with
AgentK mediation, inspect the trace, and replay the evidence. Anything outside
that claim must be clearly marked as deferred.

## Disposition Table

| Limit | v0.1 disposition | Tag blocker if |
| --- | --- | --- |
| Hosted or packaged production MCP transport | Deferred. The v0.1 operator-ready path is the local subprocess stdio proxy. | Docs or release notes imply a hosted, remote, or packaged production MCP transport exists. |
| Production signing-key custody | Deferred. v0.1 may use a configured local signing key for strict release gates, but does not claim HSM, KMS, or secret-manager custody. | A tagged release uses the static development signer, or docs imply production key custody exists. |
| Production secret storage | Deferred. v0.1 supports external secret references and metadata-only availability checks, not secret retrieval from production stores. | Docs imply AgentK can retrieve, store, rotate, or protect real production secrets. |
| Real OS sandboxing and resource isolation | Deferred. v0.1 is a policy and evidence boundary, not a process sandbox, eBPF layer, cgroup layer, or filesystem isolation runtime. | Docs imply AgentK prevents all host side effects without an external sandbox. |
| MCP resource subscriptions | Deferred until an explicit policy contract exists. Unsupported subscription methods must not become generic passthrough. | Subscription requests are forwarded without policy, provenance, and evidence. |

## Operator-Ready For v0.1

The v0.1 operator path is:

```sh
cargo run -- mcp-proxy-stdio \
  --server-id <server-id> \
  --trace-out .agentk/runs/<review-trace>.jsonl \
  --command <stdio-mcp-server> \
  --arg <server-arg>
```

This path is release-ready when it preserves these guarantees:

- lifecycle gates complete before covered MCP requests are proxied
- bounded request lines and bounded downstream waits are enforced
- AgentK-only policy metadata is stripped before forwarding
- covered tool, resource, and prompt boundaries require explicit policy
- unsupported request methods are denied instead of forwarded as passthrough
- downstream errors and malformed payloads are redacted
- trace evidence uses hashes and policy reasons instead of raw payloads
- `cargo run --locked -- release-audit` passes locally and in CI

## Not v0.1 Claims

Do not describe v0.1 as:

- production-ready
- a hosted MCP gateway
- a complete MCP protocol implementation
- a production secret manager
- a production signing-key custody system
- an OS sandbox, eBPF monitor, cgroup controller, or filesystem isolation layer
- an agent framework, planner, memory system, or orchestrator

## Release Review

Before tagging v0.1, reviewers should confirm:

- `docs/v0.1-target.md`, this document, `README.md`, `SECURITY.md`,
  `docs/threat-model.md`, and `docs/mcp-proxy.md` describe the same implemented
  behavior.
- `cargo run --locked -- release-audit` passes on the release commit.
- `AGENTK_REQUIRE_SIGNING_KEY=1` is used for strict release gates with a
  configured signing key file outside the repository.
- No generated `.agentk` traces, signing keys, private manifests, `.env` files,
  local paths, real secret references, or credential-like values are tracked.
- Release notes list the accepted limits as deferred work, not hidden caveats.

# v0.1 Release Checklist Dry Run

This is a pre-tag dry run of the signed release checklist. It is not a release
announcement and it does not create or approve a tag.

Dry run date: 2026-05-25

Base commit checked before this document was added: `b84867a`

## Result

The current tree is close to v0.1 release shape, but a tag still needs explicit
human release approval and a real maintained release signing key.

## Checks Run

- `git status --short --branch`: clean `master...origin/master`
- `gh pr list --state open`: no open PRs
- latest master CI `audit`: passing at `b84867a`
- `cargo run --locked -- readiness`: no blocking failures
- `AGENTK_REQUIRE_SIGNING_KEY=1 AGENTK_SIGNING_KEY_FILE=<temporary-outside-repo-key> cargo run --locked -- release-audit --strict`: strict gate exercised with configured signing key
- `cargo run --locked -- verify-signatures .agentk/runs/latest.jsonl --trusted-public-key <temporary-public-key>`: signatures verified with trusted-key pinning
- `git ls-files`: reviewed tracked file list
- tracked-file pattern scan for key headers, GitHub tokens, OpenAI key prefixes, `.env` references, signing-key paths, and local `/Users/` paths
- GitHub repository controls checked through `gh api`

## Passing Evidence

- Branch protection requires the `audit` status check on `master`.
- Branch protection is strict, blocks force pushes, blocks branch deletion, and
  enforces admin restrictions.
- GitHub private vulnerability reporting is enabled.
- Dependabot security updates are enabled.
- Secret scanning and push protection are enabled.
- The strict release audit accepts an owner-only configured signing key stored
  outside the repository.
- Signature verification with trusted public-key pinning succeeds for the
  dry-run trace.
- Generated `.agentk/` traces remain ignored.
- No tracked generated `.agentk/` logs, private keys, `.env` files, or signing
  key files were found.
- The tracked `.jsonl` files are public synthetic example transcripts.
- Pattern-scan hits for signing-key environment variables and key paths are
  documentation or code constants that describe safe release usage.

## Remaining Human Blockers

- A maintainer must explicitly approve the release.
- A durable release signing key must be generated or selected outside the
  repository. The dry run used a temporary key and deleted it.
- The strict release audit must be re-run on the final release commit with the
  durable release signing key.
- The signed `v0.1.0` tag must be created and verified by a maintainer.
- Release notes must repeat the accepted v0.1 limits from
  `docs/v0.1-limit-disposition.md` instead of implying production readiness.

## Accepted v0.1 Limits To Repeat In Release Notes

- The operator-ready path is the local subprocess stdio MCP proxy.
- Hosted, remote, or packaged production MCP transport is deferred.
- Production signing-key custody is deferred.
- Production secret storage is deferred.
- OS sandboxing, eBPF, cgroups, and filesystem isolation are deferred.
- MCP resource subscription flows are unsupported until an explicit policy
  contract exists.

## Next Engineering Step

The release checklist dry run does not require another strategy document. The
highest-value technical follow-up is production MCP proxy readiness within the
local subprocess stdio scope: clearer operator failure modes, realistic
interoperability transcripts, and evidence UX polish for denied boundary
crossings.
